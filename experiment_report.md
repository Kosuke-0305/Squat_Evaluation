# スクワット有効試技判定 LSTM 実験レポート

## データ概要

| 項目 | 内容 |
|---|---|
| 総レップ数（最終） | 372件（有効247 / 無効125） |
| 特徴量 | 11次元（膝角度・股関節角度・肩変位・足首変位等） |
| モデル | LSTM（可変長シーケンス対応） |

無効試技の失敗タイプ内訳：

| 失敗タイプ | 件数（初期） | 件数（最終） |
|---|---|---|
| bar_descent_score（バー下降） | 40件 | 約65件 |
| depth_score（深さ不足） | 46件 | 約46件 |
| bounce_score（バウンス） | 10件 | 約10件 |
| lockout_score（ロックアウト） | 5件 | 約5件 |
| foot_shift_score（足移動） | 1件 | 約1件 |

※ 1件のレップが複数の失敗タイプを持つ場合があるため合計は無効件数と一致しない。

---

## 実験の流れと結果

### Step 1：6出力LSTM（有効 + 失敗5種）

**設定**：全データ346件、多出力（valid + 5失敗スコア）

**結果**：
- モデルが全レップを「有効」と予測する自明解に収束
- val AUC=0.500、失敗要素 recall=0.000

**原因**：失敗スコアのアノテーション不足 + クラス不均衡（有効74%）

---

### Step 2：1:1バランス調整（有効を無効数にダウンサンプル）

**設定**：有効99件・無効99件の計198件

**結果**：
- val AUC=0.569（ランダム近傍）
- 依然として全部「有効」に近い予測

**原因**：問題設定が複雑すぎ、データ不足

---

### Step 3：2値分類に絞る

6出力から「有効/無効」の2値分類に問題を縮小。

**結果（バランスなし・全331件）**：
- val accuracy=0.742、AUC=0.490
- 全部「有効」と予測している状態と同等

---

### Step 4：特徴量・ルールベースの分析

**最小膝角度の比較**：

| | 平均最小膝角度 | 90°以下の割合 |
|---|---|---|
| 有効試技 | 92.9°（±16.5°） | 43.1% |
| 無効試技 | 102.7°（±17.2°） | 21.2% |

差=9.8°と一定のシグナルはあるが分布の重なりが大きい。

**ルールベース分類器の評価**：

```
accuracy=78%
無効 precision=0.97（誤検知はほぼなし）
無効 recall=0.32（68%を見逃す）
```

ルールベースの閾値（≤90°）が厳しすぎるため、軽微な深さ不足や
バー下降・バウンスによる無効を見逃していた。

**結論**：LSTMはこの時点でルールベースに勝てていなかった。

---

### Step 5：データ拡張

時系列データ拡張で無効試技を7倍に増強。

| 手法 | 倍率 | 内容 |
|---|---|---|
| time_warp | ×4 | 時系列を80%/90%/110%/120%にリサンプル |
| noise | ×2 | ガウスノイズ（角度±1.5°、変位±0.002）を加算 |
| lr_flip | ×1 | 左右対称に反転（角度交換・x変位符号反転） |

**拡張の対象は無効試技のみ**（有効試技を拡張してもクラス不均衡の改善にならないため）。

**結果**：無効試技 99件 → 792件（元99 + 拡張693）※後に125件→875件に更新

---

### Step 6：データリーク修正・保存基準変更

**問題点の発見**：
1. 同一レップの元データと拡張データが Train/Val に混在（データリーク）
2. `pos_weight` による損失補正が val_loss を歪め、最悪モデルを保存していた（val_loss最小時にAUC=0.513）

**修正内容**：
- 検証セットを**元データのみ**に限定
- モデル保存基準を **val_loss → val AUC** に変更
- スケジューラーも AUC ベースに変更（mode="max"、patience=8、factor=0.5）
- weight_decay=1e-4 を追加

---

### Step 7：データ品質の修正（rep_id ずれ）

**発見された問題**：

バー下降失敗のレップを含む seq ファイルで、レップ検出器がバー下降中の一時停止を
「別のレップの開始」と誤認し、1レップを2分割して記録していた。

```
seq_1_rep103_...json → reps: [rep_id=1, rep_id=2]  ← 1レップが2分割
anno_1_rep103_...csv → rep_id=103                  ← 元のレップ番号
```

rep_id が一致しないため、これらのサンプルが学習・評価から除外されていた（15件）。
これらは全てバー下降失敗（bar_descent_score=3）の無効試技であった。

**修正方法**：
- 2分割された特徴量シーケンスを連結して1レップとして統合
- rep_id を anno ファイルの番号に統一

**効果**：除外されていた無効試技 14件が学習・評価に復元（85件 → 99件）

---

### Step 8：追加データ収集（video 5）

バー下降失敗サンプルを中心に新規データ（26件）を収集・追加。
追加分にも rep_id ずれが5件存在し、同様の手順で修正。

| 項目 | 追加前 | 追加後 |
|---|---|---|
| 総レップ数 | 346件 | **372件** |
| 無効試技 | 99件 | **125件** |
| 拡張後の無効試技 | 792件 | **875件** |

---

### Step 9：5-fold 交差検証による最終評価

単一 train/val 分割では検証セット（約74件）が小さすぎてAUCの統計的ノイズが大きいため、
5-fold 層化交差検証で信頼性を確認。

**設定**：
- 元データ（372件）のみで fold 分割（StratifiedKFold）
- 拡張データ（875件）は各 fold の **train にのみ**追加
  - val に含まれる元サンプルの拡張バリアントは train から除外（データリーク防止）
- 80エポック × 5fold、GPU（NVIDIA RTX A6000）使用

**結果**：

| Fold | 検証件数（無効） | AUC |
|---|---|---|
| 1 | 75件（25件） | 0.680 |
| 2 | 75件（25件） | 0.760 |
| 3 | 74件（25件） | 0.727 |
| 4 | 74件（25件） | 0.759 |
| 5 | 74件（25件） | 0.728 |
| **平均** | | **0.731 ± 0.029** |

---

### Step 10：単一分割モデルの詳細評価（model_augmented_v4.pt）

通常学習（150エポック）した最良モデルを、学習に使っていない検証セット（74件）で評価。
チェックポイントに検証セットの base 名を保存しておくことで、再現性のある評価を実施。

**検証セット（元データのみ・74件）**：

| クラス | Precision | Recall | F1 |
|---|---|---|---|
| invalid（無効） | 0.62 | **0.80** | 0.70 |
| valid（有効） | 0.88 | 0.76 | 0.81 |
| **全体 AUC** | | | **0.780** |

- invalid recall=0.80：無効試技の80%を正しく検出できている
- invalid precision=0.62：モデルが無効と判定したうち38%は実際には有効（誤検知あり）

## 考察

### 有効だった施策

| 施策 | 効果 |
|---|---|
| 2値分類への絞り込み | 失敗スコアのアノテーション不足問題を回避 |
| データ拡張（7手法） | 無効試技の実効サンプル数を約8倍に増加 |
| データリーク修正 | 検証AUCの信頼性を確保 |
| 保存基準のAUC化 | pos_weight による損失歪みを回避し真のベストモデルを保存 |
| rep_id ずれ修正 | 除外されていた無効試技14件を復元 |
| 5-fold CV | 小さな検証セットの統計的ノイズを軽減 |

### ルールベースとの比較

ルールベースは「閾値を明確に超えた場合のみ無効」と判定するため precision は高い（0.93）が、
recall が 22% と低く、軽微な違反をほぼ見逃す。
LSTMは時系列パターン全体を学習するため、閾値では捉えにくい微妙な動作異常も検出できる。
invalid recall=0.80 はルールベース（0.22）の約3.6倍。

### 残存する課題

- **bounce/lockout/foot_shift** の件数不足（約10件・5件・1件）により、
  これらの失敗タイプはほぼ学習できていない。
  現時点の性能は主に **bar_descent と depth の識別** によって支えられていると考えられる。
- **invalid precision=0.62** は改善余地あり。
  有効/無効の境界が曖昧なレップが誤検知の主因と推測される。

### さらなる改善に必要なこと

各失敗タイプごとに **50件以上** の実データ収集が目標。
特に bounce・lockout・foot_shift を優先的に収集することで AUC=0.80 以上が期待できる。
