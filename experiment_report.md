# スクワット有効試技判定 LSTM 実験レポート

## データ概要

| 項目 | 内容 |
|---|---|
| 総レップ数（最終） | 372件（有効247 / 無効125） |
| 特徴量 | 11次元（膝角度・股関節角度・肩変位・足首変位等） |
| モデル | LSTM（可変長シーケンス対応） |

無効試技の失敗タイプ内訳：

| 失敗タイプ | 件数（初期） | 件数（最終） |
|---|---|---|
| bar_descent_score（バー下降） | 40件 | 約65件 |
| depth_score（深さ不足） | 46件 | 約46件 |
| bounce_score（バウンス） | 10件 | 約10件 |
| lockout_score（ロックアウト） | 5件 | 約5件 |
| foot_shift_score（足移動） | 1件 | 約1件 |

※ 1件のレップが複数の失敗タイプを持つ場合があるため合計は無効件数と一致しない。

---

## 実験の流れと結果

### Step 1：6出力LSTM（有効 + 失敗5種）

**設定**：全データ346件、多出力（valid + 5失敗スコア）

**結果**：
- モデルが全レップを「有効」と予測する自明解に収束
- val AUC=0.500、失敗要素 recall=0.000

**原因**：失敗スコアのアノテーション不足 + クラス不均衡（有効74%）

---

### Step 2：1:1バランス調整（有効を無効数にダウンサンプル）

**設定**：有効99件・無効99件の計198件

**結果**：
- val AUC=0.569（ランダム近傍）
- 依然として全部「有効」に近い予測

**原因**：問題設定が複雑すぎ、データ不足

---

### Step 3：2値分類に絞る

6出力から「有効/無効」の2値分類に問題を縮小。

**結果（バランスなし・全331件）**：
- val accuracy=0.742、AUC=0.490
- 全部「有効」と予測している状態と同等

---

### Step 4：特徴量・ルールベースの分析

**最小膝角度の比較**：

| | 平均最小膝角度 | 90°以下の割合 |
|---|---|---|
| 有効試技 | 92.9°（±16.5°） | 43.1% |
| 無効試技 | 102.7°（±17.2°） | 21.2% |

差=9.8°と一定のシグナルはあるが分布の重なりが大きい。

**結論**：特徴量に一定のシグナルはあるが分布の重なりが大きく、単純な閾値では高精度な分類が困難。機械学習による時系列全体の学習が必要と判断した。

---

### Step 5：データ拡張

時系列データ拡張で無効試技を7倍に増強。

| 手法 | 倍率 | 内容 |
|---|---|---|
| time_warp | ×4 | 時系列を80%/90%/110%/120%にリサンプル |
| noise | ×2 | ガウスノイズ（角度±1.5°、変位±0.002）を加算 |
| lr_flip | ×1 | 左右対称に反転（角度交換・x変位符号反転） |

**拡張の対象は無効試技のみ**（有効試技を拡張してもクラス不均衡の改善にならないため）。

**結果**：無効試技 99件 → 792件（元99 + 拡張693）※後に125件→875件に更新

---

### Step 6：データリーク修正・保存基準変更

**問題点の発見**：
1. 同一レップの元データと拡張データが Train/Val に混在（データリーク）
2. `pos_weight` による損失補正が val_loss を歪め、最悪モデルを保存していた（val_loss最小時にAUC=0.513）

**修正内容**：
- 検証セットを**元データのみ**に限定
- モデル保存基準を **val_loss → val AUC** に変更
- スケジューラーも AUC ベースに変更（mode="max"、patience=8、factor=0.5）
- weight_decay=1e-4 を追加

---

### Step 7：データ品質の修正（rep_id ずれ）

**発見された問題**：

バー下降失敗のレップを含む seq ファイルで、レップ検出器がバー下降中の一時停止を
「別のレップの開始」と誤認し、1レップを2分割して記録していた。

```
seq_1_rep103_...json → reps: [rep_id=1, rep_id=2]  ← 1レップが2分割
anno_1_rep103_...csv → rep_id=103                  ← 元のレップ番号
```

rep_id が一致しないため、これらのサンプルが学習・評価から除外されていた（15件）。
これらは全てバー下降失敗（bar_descent_score=3）の無効試技であった。

**修正方法**：
- 2分割された特徴量シーケンスを連結して1レップとして統合
- rep_id を anno ファイルの番号に統一

**効果**：除外されていた無効試技 14件が学習・評価に復元（85件 → 99件）

---

### Step 8：追加データ収集（video 5）

バー下降失敗サンプルを中心に新規データ（26件）を収集・追加。
追加分にも rep_id ずれが5件存在し、同様の手順で修正。

| 項目 | 追加前 | 追加後 |
|---|---|---|
| 総レップ数 | 346件 | **372件** |
| 無効試技 | 99件 | **125件** |
| 拡張後の無効試技 | 792件 | **875件** |

### Step 9：5-fold 交差検証による最終評価

単一 train/val 分割では検証セット（約74件）が小さすぎてAUCの統計的ノイズが大きいため、
5-fold 層化交差検証で信頼性を確認。

**設定**：
- 元データ（372件）のみで fold 分割（StratifiedKFold）
- 拡張データ（875件）は各 fold の **train にのみ**追加
  - val に含まれる元サンプルの拡張バリアントは train から除外（データリーク防止）
- 80エポック × 5fold、GPU（NVIDIA RTX A6000）使用

**結果**：

| Fold | 検証件数（無効） | AUC |
|---|---|---|
| 1 | 75件（25件） | 0.680 |
| 2 | 75件（25件） | 0.760 |
| 3 | 74件（25件） | 0.727 |
| 4 | 74件（25件） | 0.759 |
| 5 | 74件（25件） | 0.728 |
| **平均** | | **0.731 ± 0.029** |

### Step 10：単一分割モデルの詳細評価（model_augmented_v4.pt）

通常学習（150エポック）した最良モデルを、学習に使っていない検証セット（74件）で評価。
チェックポイントに検証セットの base 名を保存しておくことで、再現性のある評価を実施。

**検証セット（元データのみ・74件）**：

| クラス | Precision | Recall | F1 |
|---|---|---|---|
| invalid（無効） | 0.62 | **0.80** | 0.70 |
| valid（有効） | 0.88 | 0.76 | 0.81 |
| **全体 AUC** | | | **0.780** |

- invalid recall=0.80：無効試技の80%を正しく検出できている
- invalid precision=0.62：モデルが無効と判定したうち38%は実際には有効（誤検知あり）

## ルールベース vs LSTM 詳細比較

### ルールベース判定閾値

ルールベース分類器（`classifier.py`）は以下の閾値で各失敗要素を判定する（閾値最適化実験 Step 11 による）。
いずれか1つでも抵触した場合に無効と判定する。

| 失敗要素 | 閾値 | 判定条件 | 対象フェーズ |
|---|---|---|---|
| 深さ不足 | ≤ 110° | シーケンス中に代表膝角度をこの値以下まで曲げなければ抵触 | 全体 |
| ロックアウト不足 | ≥ 167° | 開始・終了付近3フレームで代表膝角度がこの値以上でなければ抵触 | 開始・終了 |
| バー下降 | ≥ 0.010 | 肩 Y delta が前フレームより増加すれば抵触 | 上昇フェーズ |
| 反復動作 | ≥ 10° × 2回 | 股関節角度の方向逆転が2回以上で抵触 | 下降フェーズ |
| 足ずれ | ≥ 0.05 | 足首 X delta の絶対値がこの値以上のフレームがあれば抵触 | 全体 |

### クラス別 Precision / Recall 比較

**ルールベース（閾値最適化後・全372件）**：

| クラス | Precision | Recall | F1 |
|---|---|---|---|
| invalid（無効） | 0.71 | 0.56 | 0.62 |
| valid（有効） | 0.80 | 0.88 | 0.84 |
| **AUC** | | | **0.721** |

**LSTM（val 74件・学習済みモデル）**：

| クラス | Precision | Recall | F1 |
|---|---|---|---|
| invalid（無効） | 0.62 | 0.80 | 0.70 |
| valid（有効） | 0.88 | 0.76 | 0.81 |
| **AUC** | | | **0.780** |

### 解釈

**ルールベースの特徴**：
- invalid precision=0.71：無効と判定したうち71%が正しい（誤検知は29%）
- invalid recall=0.56：実際の無効試技の44%を見逃す
- 固定閾値のため、軽微な違反や複合的な失敗パターンの検出に限界がある

**LSTMの特徴**：
- invalid recall=0.80：ルールベース（0.56）より高い検出率
- invalid precision=0.62：検出精度は下がる（有効を無効と誤判定するケースがある）
- 時系列パターン全体を学習することで、閾値では定義しにくい微妙な動作異常を捉えられる
- 境界が曖昧なレップ（unanimous=0）の判定が難しく、precision 低下の主因と推測される

**用途別の使い分け**：
- **確実な違反だけを指摘したい場合**：ルールベース（precision 重視）
- **見逃しを減らしたい場合**：LSTM（recall 重視）

### Step 11：ルールベース閾値最適化実験

ルールベース分類器の閾値が最適かどうかを検証するため、閾値スウィープ実験を実施。

#### 発見されたバグ

`classify_attempt()` 内の `_repr_knee()` 関数が、関節未検出フレーム（値 = 0.0）に対して
0.0 を返す実装になっていた。depth チェックは `any(ka <= 90)` であるため、
0.0 ≤ 90.0 が常に True となり、**depth_score が全サンプルで発火しない**という致命的なバグが存在した。

実際のデータ確認（0 を除外した膝角度最小値の分布）：

| | 有効（修正後） | 無効（修正後） |
|---|---|---|
| 平均最小膝角度 | 92.3°（±17.0°） | 100.9°（±16.7°） |
| 90° 以下の割合 | 43.3%（107件） | 24.0%（30件） |

有効試技でも56.7%が90°に未到達であり、90°の閾値は厳格すぎることが判明。

#### depth 閾値スウィープ（バグ修正後）

| 閾値 | AUC | inv_Prec | inv_Rec | 発火件数 |
|---|---|---|---|---|
| 90° | 0.635 | 0.43 | 0.84 | 235 |
| 95° | 0.671 | 0.47 | 0.79 | 195 |
| 100° | 0.682 | 0.52 | 0.69 | 148 |
| **105°** | **0.687** | 0.58 | 0.58 | 104 |
| **110°** | **0.693** | 0.68 | 0.50 | 68 |
| 115° | 0.664 | 0.74 | 0.40 | 44 |

depth 閾値 110° で AUC が最大。これは「110° まで膝を曲げなかった場合を深さ不足と判定」する設定であり、
わずかな屈曲しかしていない浅いスクワットを検出する。

#### その他の閾値スウィープ

**lockout 閾値（現在 165°）**：167〜170° に緩和すると発火件数が増え AUC が 0.643 に微増。

**bar_descent 閾値（現在 0.03）**：0.01 に厳格化すると発火 10 件（+6）、AUC が 0.647 に微増。

**foot_shift 閾値**：どの閾値でも発火件数 = 0。このデータセットには足ずれサンプルが実質的に存在しない。

#### 最終比較（全372件）

| 設定 | AUC | inv_Prec | inv_Rec | inv_F1 |
|---|---|---|---|---|
| depth=110°のみ最適化 | 0.693 | 0.68 | 0.50 | 0.58 |
| 全閾値最適化 | **0.721** | 0.71 | 0.56 | 0.62 |
| **LSTM 5-fold CV** | **0.731 ± 0.029** | — | ~75% | — |

**最適閾値（depth=110°, lockout=167°, bar_descent=0.010）での詳細分類レポート（全372件）:**

```
              precision    recall  f1-score   support

  invalid(0)       0.71      0.56      0.62       125
    valid(1)       0.80      0.88      0.84       247

    accuracy                           0.77       372
   macro avg       0.75      0.72      0.73       372
weighted avg       0.77      0.77      0.77       372

AUC: 0.721
```

**閾値最適化後のルールベースは AUC=0.721 となり、LSTM（0.731）に迫る性能を発揮する。**

ただし以下の点に注意：
- この最適化はテストデータと同じ372件で閾値を決定しているため過学習リスクあり
- LSTM は独立した検証セット（74件）で 0.780 を達成しており、純粋な汎化性能はLSTMが優位
- Precision/Recall のバランスが異なる（ルールベース最適: Prec=0.71, Rec=0.56 / LSTM: Prec=0.62, Rec=0.80）
- foot_shift はデータセット内で発火件数 = 0（足ずれサンプルが実質的に存在しない）

## 考察

### 有効だった施策

| 施策 | 効果 |
|---|---|
| 2値分類への絞り込み | 失敗スコアのアノテーション不足問題を回避 |
| データ拡張（7手法） | 無効試技の実効サンプル数を約8倍に増加 |
| データリーク修正 | 検証AUCの信頼性を確保 |
| 保存基準のAUC化 | pos_weight による損失歪みを回避し真のベストモデルを保存 |
| rep_id ずれ修正 | 除外されていた無効試技14件を復元 |
| 5-fold CV | 小さな検証セットの統計的ノイズを軽減 |

### ルールベースとの比較

閾値最適化後のルールベース（AUC=0.721）と LSTM 5-fold CV（AUC=0.731）はほぼ同等の AUC だが、
Precision/Recall のバランスが異なる。ルールベースは precision=0.71 で誤検知を抑えつつ recall=0.56。
LSTM は recall=0.80 と見逃しを大幅に減らす一方、precision=0.62 でやや誤検知が多い。
LSTM の独立検証セット AUC=0.780 はルールベース（0.721）を上回り、汎化性能でも優位。

### 残存する課題

- **bounce/lockout/foot_shift** の件数不足（約10件・5件・1件）により、
  これらの失敗タイプはほぼ学習できていない。
  現時点の性能は主に **bar_descent と depth の識別** によって支えられていると考えられる。
- **invalid precision=0.62** は改善余地あり。
  有効/無効の境界が曖昧なレップが誤検知の主因と推測される。

### さらなる改善に必要なこと

各失敗タイプごとに **50件以上** の実データ収集が目標。
特に bounce・lockout・foot_shift を優先的に収集することで AUC=0.80 以上が期待できる。
